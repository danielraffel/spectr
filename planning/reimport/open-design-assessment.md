# Open Design (nexu-io/open-design) — adapter feasibility assessment

**Repo:** https://github.com/nexu-io/open-design (Apache-2.0, TypeScript, ~76 MB checkout, very
active — 2026-04-28 → 2026-05-03 last-touched at audit time).

**Pitch:** local-first open-source alternative to Anthropic's Claude Design. Auto-detects 12
coding-agent CLIs on `$PATH` and drives them via 31 design skills + 72 design systems. Output
artifacts stream as `<artifact identifier="…" type="…" title="…">…</artifact>` blocks.

## What it produces (the bits we'd ingest)

| Artifact kind         | Output shape                                           | Inner format                                                          |
|-----------------------|--------------------------------------------------------|-----------------------------------------------------------------------|
| `html`                | Self-contained HTML doc with inline `<style>` + `<body>` markup | CSS custom properties (`--bg`, `--accent`), semantic class system, plain HTML5 |
| `react-component`     | Single .jsx / .tsx file                                | React 18 functional component, inline styles or CSS-in-JS             |
| `svg`                 | Single `<svg>...</svg>`                                | path / rect / text                                                    |
| `mini-app`            | Self-contained HTML with embedded JS                   | Bundled React + Babel-standalone (same shape as Claude Design exports)|
| `markdown-document`   | Markdown                                               | Standard markdown                                                     |
| `design-system`       | JSON                                                   | Tokens + component recipes                                            |
| `deck`                | HTML deck (Reveal-style)                               | One page per slide                                                    |

The **artifact-level** identifier is stable per-stream, but the **inner DOM** is fresh-generated
each run — same problem as Claude Design.

## Stable identifier story (the part that matters for reimport)

- `<artifact identifier="…">` — yes, exists; agent-supplied, recommended UUID-shaped, used for
  preview-pane routing and incremental streaming reconciliation
  (`apps/web/src/artifacts/parser.ts`).
- Inner-node IDs — none. Plain HTML / JSX, no `data-od-id` / `data-stable-id` attribute
  convention. The skills' `references/layouts.md` does encourage `class="hero"` / `class="logo"`
  semantic naming, but classnames are **not unique** within an artifact.

For a reimport-safe pipeline we'd need either:

1. **Adapter-side derived stable IDs** — hash of (parent path + tag + role + ordinal-within-role
   + first-3-words-of-text). The same heuristic could be used for any unstamped source (Claude,
   Stitch, v0). Drift behavior is well-understood: rename → new id, reorder same-role siblings →
   same id, edit text in middle node → same id (because role + ordinal preserved, text only
   used for tie-break).
2. **Upstream change** — get open-design to mint and emit `data-od-id="..."` on every element
   the skill renders. Skills would have to participate; this is non-trivial because they're
   prompted, not generated by typed templates.

(1) is feasible today and is the recommended path. We should NOT block on (2).

## Adapter feasibility — open-design → Pulp canonical AST

| Step                                  | Feasibility | Notes |
|---------------------------------------|-------------|-------|
| Parse `<artifact>` envelope           | Easy        | Vendor open-design's parser.ts shape (it's 150 LOC, MIT-style) into Pulp's import path |
| Route by `type=html` → existing Stitch HTML pipeline | Easy | `parse_stitch_html` already handles inline `<style>` + `<body>` markup |
| Route by `type=react-component` → v0 path | Medium | `parse_v0_tsx` exists but is JSX-not-bundle; needs bundler invocation for runtime IR |
| Route by `type=mini-app` → Claude bundle path | Easy | `parse_claude_html_with_runtime` already handles inline `<script type="text/babel">` |
| Route by `type=svg` → new path | Easy | One SvgPath widget per `<path>` |
| Tokens: lift `:root { --... }` to W3C tokens | Already works | Stitch parser does this |
| Stable IDs (heuristic) | Medium | Requires new IR pass — not source-specific, applies everywhere |
| Design-system JSON ingestion | Medium | Format is documented in `design-systems/` skills; recipes need a Pulp Theme mapping pass |

## Gaps that block reimport-safe ingestion

1. **No inner-node stable IDs** — same as every other source. Address via a generic
   `core/view/src/design_import_id_pass.cpp` that runs after every adapter.
2. **Streaming output, not file** — open-design emits artifacts via a daemon over WebSockets.
   We'd want a `pulp import-design --from open-design --artifact-id <id>` that reads the
   daemon's exported artifact file (`apps/daemon/src/artifact-manifest.ts` writes them to disk
   per-project). Or accept stdin so an agent can pipe.
3. **`type=react-component` has no bundler** — Pulp would need to invoke esbuild on the
   single-file component before walking it through the runtime path. Same shape as #995's
   "build the React bundle" gap.
4. **Design systems are JSON recipes, not WCAG tokens** — needs a 2nd-tier converter
   (open-design design-system.json → Pulp Theme).

## Minimal-changes proposal

Adapter-side only. No upstream changes to open-design.

1. Add `DesignSource::open_design` to `design_import.hpp` enum.
2. Add `parse_open_design_artifact(const std::string& html_or_jsx, OpenDesignArtifactType type)`
   that routes by artifact type to the existing parser (claude/stitch/v0).
3. Add an `artifact-envelope` pre-pass that strips `<artifact …>…</artifact>` wrappers and
   surfaces `identifier` / `type` / `title` as IR root metadata.
4. Add the generic stable-ID heuristic pass (path+tag+role+ordinal+text-tie-break) — applies to
   every existing source, not just open-design.

Total estimated LOC: ~400 in C++, mostly in the new envelope/heuristic passes.

## Recommendation

**Treat open-design as a first-class peer of Claude Design** — same mental model, same input
shape, MIT-/Apache-licensed, growing fast, AGPL-incompatible-free, on-disk artifact output that
fits Pulp's "manual file export" pattern. The adapter is small and reuses 90% of the existing
HTML / runtime pipelines.

The work that matters is **the generic stable-ID + reimport-safe scaffolding** that benefits
every source. open-design is a pull-in target for that work, not a special-cased branch of it.
