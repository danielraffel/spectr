# Repo audit — pulp/import-design pipeline (read-only, 2026-05-01)

## TL;DR

Pulp's `pulp import-design` is a **one-shot, generate-then-forget** pipeline today. Nothing in
`core/view/src/design_import.cpp` (2902 LOC) or `tools/import-design/pulp_import_design.cpp`
(550 LOC) supports a second run that preserves DSP bindings, owned files, or downstream edits.
Re-running the CLI overwrites `ui.js` / `tokens.json` / `classnames.json` / `bridge_handlers.cpp`
with no diff, no merge, no warning.

`compat.json` (461 LOC, just landed via #1166/#1027) is a **passive** reference document. It
does not gate the import path or surface gaps in the generated output.

## Findings

### 1. ID assignment is positional, not stable (CRITICAL for reimport + DSP binding)

`core/view/src/design_import.cpp:1828-1833` and `:2059-2061`:

```cpp
static void generate_node(std::ostringstream& ss, const IRNode& node, …
                           int& var_counter, const std::string& parent_var) {
    std::string var = sanitize_var(node.name.empty() ? node.type : node.name);
    if (depth > 0) var += std::to_string(var_counter++);
    else var = opts.root_variable;
```

IDs are sequential indices over the IR walk. A node added in the middle of a sibling list shifts
**every following ID by +1**. The same is true in the native generator at `:2381-2387`. Stable IDs
are explicitly absent — the IR (`design_import.hpp:103-116` `IRNode`) has no `id` / `stable_id` /
`source_id` field at all. Only `name`, `type`, and `attributes`.

**Impact:** Any reimport silently re-labels nodes. A `Knob_3` that maps to the `cutoff` parameter
on import-1 becomes `Knob_4` on import-2 if the designer added a sibling — the binding by
widget-id-as-param-name (see #2 below) breaks with no error.

### 2. DSP binding by widget-id-as-param-name string match (FRAGILE)

`core/view/src/widget_bridge.cpp:754-783, 1260-1290`:

```cpp
void WidgetBridge::sync_from_store() {
    …
    for (size_t i = 0; i < store_.param_count(); ++i) {
        auto* info = &store_.all_params()[i];
        if (info->name == widget_id) {
            knob->set_value(store_.get_normalized(info->id));
            break;
        }
    }
}
```

`getParam(name)` / `setParam(name, value)` similarly look up params by the widget's string id.
There is no per-widget binding ledger, no preserved id-mapping, and no warning when a previously
bound widget id disappears.

**Combined with finding #1:** every reimport silently severs every DSP binding whose widget moved
in the source design.

### 3. No snapshot / lockfile / generated-vs-owned boundary

There is no:
- `.pulp-import.json` or equivalent that records the previous import's IR + output paths +
  fingerprint of the input HTML.
- File header banner like `/* @generated by pulp import-design — do not edit */` on emitted JS.
- Owned-files list (`bridge_handlers.cpp`, `CMakeLists.txt`, hand-edited token overrides).
- Drift detection (does the generated `ui.js` still match what we'd emit from the same input
  today?).

Re-running clobbers everything in the output directory and lower-cases what should be a
controlled merge into "generated, do not touch" vs "owned, hand-edited".

The bridge scaffold (`render_claude_bridge_scaffold` in `core/view/src/design_import.cpp`) is
the only file marked as a starter — and even that gets overwritten on every run unless the user
remembers to pass `--no-bridge-scaffold`.

### 4. compat.json is not consulted by the import path

`compat.json` is a 461-line static reference that documents 250+ CSS / RN / Yoga / React
surface props with `status` / `mapsTo` / `supportedValues` / `unsupportedValues`. It was
generated by walking three reference specs and four implementation surfaces (#1166 audit).

**It is not loaded by `pulp import-design` at runtime.** No code path in
`tools/import-design/pulp_import_design.cpp` or `core/view/src/design_import.cpp` reads or
references `compat.json`. The CLI does not classify imported props by status, does not warn on
`status: missing` props, does not refuse to import on N+ critical-prop drops, and does not emit
an `import-report.json` per #1027's acceptance criteria.

This means a Claude/Stitch import with 47 `boxShadow` instances (currently `status: supported`
but `partial` for spread-radius) and 12 `marginInlineStart` instances (currently
`status: missing`) ships generated JS that silently no-ops on the missing props. The user
discovers it only via the screenshot diff.

### 5. Versioned source detection (#1031) lives in compat.json schema, not in the parser

`compat-schema-version: "0.1"` is at the top of `compat.json`, but neither
`parse_claude_html`, `parse_stitch_html`, `parse_figma_json`, `parse_v0_tsx`, nor
`parse_pencil_json` does any fingerprint-based version detection. The CLI selects a parser
purely on `--from <source>`. There is no `--detect-only`, no `--report-new-format`, no
fallback-with-warning behavior on unknown export shapes.

### 6. `--execute-bundle` runtime path improves coverage but doesn't address reimport

`parse_claude_html_with_runtime` (`design_import.hpp:215`) evaluates the React bundle in a
headless `ScriptEngine` and walks the materialized DOM. This raises the IR fidelity past the
9-element loader-shell floor — but the IR walker still emits IDs by `var_counter`, so two runs
of `--execute-bundle` against the same HTML produce identically-numbered output, while two runs
against slightly-edited HTML produce divergent IDs. Same fragility as the static path.

### 7. Asset extraction is half-pipelined

- `parse_claude_bundle` (`design_import.hpp:186`) returns `assets[]` with uuid + mime + bytes.
- `extract_claude_classnames` (`design_import.hpp:251`) returns the cascade-merged classname map.
- The CLI emits `classnames.json` (#1035) for Claude only.

But: the asset list itself is never emitted to disk. There is no `assets/` directory written
next to `ui.js`. There is no asset URL → bridge image-load mapping. Fonts (woff2) and inline
SVGs in the bundle are dropped on the floor unless the runtime path happens to materialize
them.
